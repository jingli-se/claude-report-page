02

对话要点 / Conversation Key Points (from pasted thread)

背景确认 · Background confirmed

CPA模型目前少学习约25%的激活样本。原因是A2W2A流量没有配置MMP点击链路，Adjust无法对这部分流量归因，actv 信号无法进入模型训练。Temu确认其内部真实转化率与模型转化率差距约25%。

The CPA model is under-learning ~25% of activation signals. A2W2A traffic has no MMP click link configured → Adjust cannot attribute it → actv is excluded from training.

去重前提疑问 · Dedup prerequisite challenged

Temu（cris.chen）提出：S2S的actv改名为install后仍走S2S，不经Adjust——去重的基础在哪？ Liftoff（Yifan）回应：去重必须依赖Adjust的install信号，因此A2W2A侧必须先配置MMP点击链路。而Temu表示这部分"还在评估中"，造成进展阻塞。

"de-duplication 必须依赖adjust的install信号，所以没太理解这里为什么还有评估的事情？"
方案扩展讨论 · Scope expansion raised

Robert提出：希望在MMP计划中也做测试，将部分 actv 改名为 install 回传给Liftoff，验证去重是否生效，同时解决MMP侧的25%缺口。这是推动双方讨论并集方案的初始动因。

"今天早上Robert老师也想问你们mmp的一部分campaign要不要给一些actv改成install的信号我们看看，这样MMP的campaign那些也不耽误，同时看dedup的是不是work"

Yifan's interpretation: Robert wants to test dedup on MMP campaigns, and the actv+install union is the proposed solution direction to address the 25% under-learning issue.

最新更新 · Latest update — Temu's two explicit options

Temu（cris.chen）针对MMP计划的actv+install并集，明确提出两个方向，并强调不希望用"改名"方式破坏现有信号体系：

"1. 直接你方将mmp里的actv信号当中install信号使用做去重，这样我方就不用做任何调整就可完成测试；2. 如果你们不想改，我们不希望使用改名的方式做这件事，而是多回传一路install信号给你们，这样不会破坏现状"

Option 1: Liftoff internally treats MMP actv as install for dedup — zero Temu changes needed. Option 2: Temu sends an additional separate install postback alongside actv on MMP campaigns.

03

Temu新提方案 / Temu's Two Proposed Options (Latest)

Latest · 最新提议 — MMP Campaign scope · Temu明确不希望用改名方式

A
Liftoff内部映射 / Liftoff absorbs actv

由 Liftoff直接将MMP计划中收到的 actv 当作 install 使用，在Liftoff侧完成映射和去重逻辑。

Temu无需做任何改动，可立即开始测试。

Liftoff internally maps MMP actv → install. Zero changes on Temu's side — fastest path to test.

B
Temu额外回传 / Temu sends extra signal

由 Temu在MMP计划中额外多回传一路 install 信号给Liftoff，与 actv 并存，不对现有事件做任何改名。

不破坏现状，但需Temu新增一路回传集成。

Temu sends a new parallel install postback alongside actv on MMP campaigns. Preserves current signal integrity.
